Límite y supuestos

Esto zanja un schema de contenido estable para eventos y reformas (y lo mínimo que necesitan: efectos/modificadores y diccionarios base). La decisión es deliberadamente “simple + validable” para que el contenido no crezca torcido.

Decisiones cerradas
1) Formato y estructura de “Content Pack”

Formato: JSON únicamente (no YAML).
Unidad de entrega: un “pack” versionado con manifest.json + archivos de datos.

Estructura mínima:

content/
  manifest.json
  strings/
    es.json
    en.json              (opcional)
  core/
    regions.json
    igs.json
    movements.json
  rules/
    target_config.json
    aggregation_config.json
    legislative_config.json
  templates/
    effects.json
    events.json
    reforms.json

2) IDs estables (convención obligatoria)

snake_case ASCII, sin espacios.

Prefijos por tipo:

eff_* effects

evt_* events

ref_* reforms

mov_* movements

ig_* IGs

Regiones: IDs sin prefijo (p.ej. metropolitana) o reg_*. Decisión: sin prefijo (más legible en paths regions.metropolitana.*).

Prohibido renombrar IDs: si cambia, se hace alias map (ver validación).

3) Localización (sin strings hardcodeadas)

Todo texto visible usa claves de localización, no texto embebido.

Archivo por idioma: strings/es.json con diccionario { "key": "texto" }.

Claves recomendadas:

event.evt_media_scandal.title

event.evt_media_scandal.desc

event.evt_media_scandal.opt.minimize

reform.ref_labor.title

etc.

4) Unidades numéricas (determinismo)

Todo valor que toque el estado usa fixed-point int:

Métricas 0–100: 0..10000 (S=100)

Aprobación -100..+100: -10000..+10000

Multiplicadores: factorS con base 10000 = 1.00

En contenido no se permiten floats. Solo int.

Schema cerrado (campos obligatorios)
A) manifest.json
{
  "content_pack_id": "base_chile_fictional",
  "content_pack_version": 1,
  "content_schema_version": 1,
  "min_game_schema_version": 1,
  "languages": ["es"],
  "default_language": "es",
  "files": {
    "core/regions.json": "sha256:...",
    "core/igs.json": "sha256:...",
    "templates/effects.json": "sha256:...",
    "templates/events.json": "sha256:...",
    "templates/reforms.json": "sha256:..."
  }
}

B) core/regions.json

Raíz:

{ "regions": [ RegionDef... ] }

RegionDef (obligatorio):

{
  "id": "metropolitana",
  "name": "Metropolitana",
  "weight_ppm": 62500,
  "macrozone": "CENTER",

  "admin_capS": 5000,
  "industry_capS": 5000,
  "extractive_capS": 5000,
  "social_capS": 5000,
  "populationS": 5000
}

Campos requeridos:
id, name, weight_ppm, macrozone

Campos opcionales (0..10000; default recomendado 5000):
admin_capS, industry_capS, extractive_capS, social_capS, populationS

Restricciones:
- Σ weight_ppm = 1_000_000.
- macrozone ∈ { "NORTH","CENTER","SOUTH","AUSTRAL" }.
- Los campos *_capS y populationS no deben cambiar durante una campaña; son “data estática” del pack (se versiona por content_pack_version).

Validación (obligatoria) — reglas cerradas
1) Validación estructural (JSON Schema)

Cada archivo tiene su jsonschema (v2020-12 o similar).

Falla si falta un campo requerido, tipo incorrecto, enum inválido.

2) Validación semántica (lint del pack)

Debe fallar si:

IDs duplicados por tipo.

Referencias rotas:

template_id inexistente (effects).

event_id inexistente en followups.

ig_* inexistente en igs_stance.

target no existe en TargetConfig (incluye internals.* y wildcards permitidos).

Claves de localización inexistentes en strings/{lang}.json.

Rangos inválidos:

valueS fuera de clamps del target (cuando sea determinable).

duration_weeks <= 0, cooldown_weeks < 0, etc.

Uso inválido del DSL:

cmp sin valueS, o usando value float.

scope=REGION sin vars.region (o sin región resoluble).

max_per_campaign = 0 (se considera inválido: usar disabled=true si se quiere apagar).

3) Compatibilidad de contenido

manifest.min_game_schema_version debe ser <= versión del ejecutable.

content_schema_version se migra con scripts de contenido (separado de migración de save).

Herramientas mínimas (zanjadas)

validate_content (CLI)

validate_content content/:

valida JSON Schema

valida semántica

Riesgos/ambigüedades

Si permites “scripting” en condiciones (expresiones libres), el schema deja de ser estable y la validación se vuelve difícil. Por eso se fija DSL mínima.

Si se permiten strings embebidas, luego migrar a localización es costoso; por eso se prohíbe desde el inicio.
