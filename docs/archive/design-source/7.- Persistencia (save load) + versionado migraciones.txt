2) Formato de archivo (container) y layout interno

Decisión: un único archivo “contenedor” (tipo zip) con JSON dentro.

Extensión: *.juego-save (o *.save si prefieres).

Contenido mínimo dentro del contenedor:

meta.json (cabecera liviana)

state.json (GameState completo)

integrity.json (hashes/checksums)

(opcional) last_report.json (último TurnReport ya renderizado, para abrir el save y mostrar “lo último que pasó” sin recalcular UI)

Por qué: es fácil de versionar, inspeccionar, comprimir y migrar por JSON; y permite agregar archivos sin romper formato.

3) Versiones: 3 niveles (separados)

Para evitar “romper campañas” por mezclar cambios, se versiona por capas:

save_format_version (del contenedor): cambia raramente (p.ej., si dejas de usar zip o cambias layout).

schema_version (del GameState): es el meta.save_version del estado. 

1.-Modelo de estado canónico + …

content_version (pack de contenido): versión del set de plantillas/eventos/reformas y diccionarios core (regions/igs/etc.).

Regla: cambios “solo de contenido” (incluye agregar campos estáticos en core/regions.json como admin_capS/industry_capS/...):
- incrementan content_pack_version
- NO requieren migración de schema_version (GameState) si esos campos no se guardan en el save (se rehidratan desde el pack)
- pueden gatillar warning o best-effort si content_pack_hash no coincide (según política del punto 3 y riesgos)

4) Política de compatibilidad (regla simple y estricta)

Backward compatible: el juego actual debe poder cargar saves con schema_version <= current_schema_version aplicando migraciones secuenciales.

Forward incompatible: si schema_version > current_schema_version, se rechaza (mensaje claro).

Ventana de soporte: “N versiones mayores” (decisión: N=2). Saves más antiguos requieren “converter” externo o se rechazan.

5) Migraciones (pipeline zanjado)

Decisión: migraciones puras sobre JSON, en cadena, 1 paso por versión.

Tabla/registro en código: Migration(fromVersion, toVersion, apply(JsonRoot) -> JsonRoot)

Loader:

lee state.json

obtiene schema_version

mientras schema_version < CURRENT aplica Migration[schema_version]

valida invariantes (clamps, ids, rangos)

termina con schema_version == CURRENT

Reglas de migración (para mantener estabilidad):

No se re-balancea nada en migración (no “tuning”); solo transformaciones estructurales / renombres / defaults.

Cada migración debe ser idempotente si se ejecuta una sola vez (no depende del orden fuera de la cadena).

Cuando se agregan campos nuevos: defaults deterministas.

Cuando se renombra un ID: usar alias map (ver punto 6).

6) Estrategia para cambios de IDs (evitar romper saves)

Decisión: mantener “alias maps” por dominio (regiones, eventos, reformas, IGs).

Ejemplo: region_aliases: { old_id -> new_id } aplicado en load/migración.

Esto está alineado con el riesgo ya identificado: cambiar regiones/IDs tarde rompe saves; mitigación propuesta: alias map en loader.

Riesgos/ambigüedades

Cambios de contenido (plantillas/reformas) con saves existentes: si no se fija content_pack_hash, pierdes determinismo; si se fija demasiado estricto, frustras iteración. Mitigación: content_pack_hash estricto + modo “best-effort” con warning.

Migraciones acumuladas: si se dejan crecer sin disciplina, el loader se vuelve frágil. Mitigación: ventana N=2 + “consolidación” ocasional (export/import a schema nuevo).

Corrupción de archivo: mitigado con hash + backup rotativo, pero igual puede ocurrir si se corta energía durante write (de ahí el write atómico).

Si implementas exactamente este contrato, queda “listo” en el sentido técnico: puedes programar save/load sin volver a rediseñar el motor después.
