2) Modelo de causas (CauseRef) y cadena causal

Cada contribución se atribuye a una CauseRef con identidad estable:

CauseRef:

category: DECISION | EVENT | MODIFIER | SYSTEM

id: string (id del contenido o del sistema)

parent: CauseRef? (opcional; para cadena causal)

Regla de cadena:

Un EffectInstance (MODIFIER) siempre tiene parent apuntando al DECISION/EVENT que lo originó (cuando aplique).

Cambios internos “ocultos” que afecten visibles deben registrar causa atribuible (evento/decisión/modificador).

3) Único punto de escritura: API de cambio con contexto

Para evitar “cambios invisibles”, se impone una regla de motor:

Cualquier mutación a un target visible debe pasar por:

ApplyModifier(target, delta/factor/set, causeRef) o

SetTargetValue(target, value, causeRef)

Prohibido (a nivel de disciplina de código): mutar diccionarios del estado directamente para targets visibles.

Esto garantiza la coherencia exigida por el diseño (si cambia algo visible, hay causa).

4) Cómo se calcula el delta y cómo se reparte por causa (sin “magia”)
4.1 Regla general: el log debe cuadrar exacto con el resultado final

Para cada target en un tick, debe cumplirse:

x_end - x_start = Σ(contribuciones por causa)

Si hay clamps, normalizaciones o rounding, se registran como causas SYSTEM para cerrar la contabilidad.

4.2 Aplicación con logging (por target, por tick)

Se define un orden determinista (compatible con el motor de efectos):

Caso A: hay SET activo sobre el target

Se elige el SET ganador (por priority, tie-break determinista).

Se aplica x := clamp(set_value).

Se registra una sola entrada causal:

delta = x - x_start con causa = (MODIFIER del efecto SET, o SYSTEM si fue una acción directa de sistema)

Caso B: no hay SET

ADD: se aplican por causa, exacto:

para cada causa c: x += add_c y loggea add_c a c

MUL: como no es lineal, se aplica secuencialmente (orden determinista por priority, luego effect_instance_id) para que el reparto sea exacto:

para cada factor f con causa c:

x_new = round(x * f / base)

loggea delta = x_new - x a causa c

x = x_new

Clamp final (si corresponde por TargetConfig):

si x se recorta, loggea residuo a causa SYSTEM:CLAMP

Rounding residual (si tu fixed-point hace “half up”):

si queda diferencia por rounding (puede ocurrir en productos encadenados), se loggea a SYSTEM:ROUNDING

4.3 Normalizaciones obligatorias con logging

Ejemplo crítico: igs.*.clout se renormaliza para sumar 100.

Cada ajuste de renormalización se loggea como:

causa SYSTEM:IG_CLOUT_NORMALIZE

delta por cada IG afectado (positivos/negativos)
Esto es necesario para que el reporte no muestre “cambios fantasmas” sin causa atribuible.
