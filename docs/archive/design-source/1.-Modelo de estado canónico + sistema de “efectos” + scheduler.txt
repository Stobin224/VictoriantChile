3) Contrato del sistema de efectos

Un “efecto” en runtime es una instancia que aplica uno o más modificadores por tick (o instantáneo).

3.1 EffectInstance

Campos obligatorios:

id: string (uuid o incremental)

template_id: string (o embebido si no hay templates aún)

source: SourceRef

type: DECISION|EVENT|REFORM|SYSTEM

key: string (id del contenido)

start_tick: int

end_tick_exclusive: int | null (null = infinito)

stack_key: string (para resolver stacking)

stack_mode: STACK | REPLACE | REFRESH | MAX | STACK_LIMIT_N

stack_limit_n: int? (solo si STACK_LIMIT_N)

priority: int (entero; default 0)

magnitudeS: int (escala; default 100*S para “1.00”; permite reutilizar templates)

3.2 EffectTemplate (contenido)

Campos:

id, name, tags[]

mods[]: ModifierSpec

3.3 ModifierSpec (lo que cambia)

Cada mod define:

target: TargetPath (string, p.ej. metrics.legitimidad, igs.sindicatos.approval, regions.rm.support)

op: ADD | MUL | SET

valueS: int

ADD: delta por tick en escala S

MUL: factorS (base = 100*S)

SET: valor absoluto en escala S

is_per_tick: bool

true: se aplica cada tick mientras esté activo

false: se aplica una sola vez al activarse (instantáneo)

clamp_minS, clamp_maxS: int? (opcional; si no, usa clamps globales del target)

Regla: el motor no adivina clamps; cada target tiene clamps globales en una tabla TargetConfig.

4) Reglas de stacking (cerradas)

Al aplicar una nueva EffectInstance con stack_key:

STACK: se agrega; convive con otras instancias mismo stack_key.

REPLACE: elimina todas las instancias activas con stack_key y agrega la nueva.

REFRESH: si existe alguna activa con stack_key, solo actualiza su end_tick_exclusive al de la nueva (y opcionalmente su magnitude al de la nueva). No crea otra instancia.

MAX: si existe alguna activa con stack_key, mantiene una sola:

Para efectos ADD: gana la de mayor abs(delta_total_por_tick) (si empatan, gana la de mayor priority, luego id).

Para MUL: gana la de mayor desviación desde 1.00 (abs(factorS - 100*S)).

STACK_LIMIT_N: se permite hasta N instancias; si excede, se expulsan las más antiguas (menor start_tick, luego id).

5) Orden de aplicación por tick (scheduler)

Para ejecutar 1 tick:

Tick(t → t+1):

tick += 1 y actualiza date (si se usa calendario).

Expirar: elimina efectos con end_tick_exclusive != null && end_tick_exclusive <= tick.

Aplicar instantáneos de inicio:

Para cada efecto cuya start_tick == tick, aplica mods con is_per_tick=false (una vez).

Aplicar mods per-tick:

Construye una lista de ModifierRuntime activos con is_per_tick=true y start_tick <= tick < end.

Agrupa por target.

Para cada target, computa el update con estas reglas (deterministas):

5.1 Regla matemática por target (cerrada)

Sea x el valor actual del target (int escala S).

Si existe al menos un SET activo sobre el target:

Selecciona uno: el de mayor priority, si empata el menor effect_instance_id lexicográfico.

Define x_new = clamp(SET.valueS, min, max).

Ignora todos los ADD/MUL para ese target en ese tick.

Si no hay SET:

add_total = suma(ADD.valueS) (de todas las instancias activas)

mul_total = producto(MUL.factorS) con escala (ver abajo)

delta_mul = round( x * (mul_total - base) / base ) donde base = 100*S

x_new = clamp( x + add_total + delta_mul, min, max )

Producto escalado determinista (enteros):

mul_total = base

por cada f en factores:

mul_total = round(mul_total * f / base) usando rounding “half up”.

Normalización posterior obligatoria (una vez por tick):

IG.clout se renormaliza para que la suma sea exactamente 100*S (ajuste por residuo al mayor clout).

Clamps finales por cada target (si algún target no clamp-eó antes).

Fin tick: emite al log causal (ver abajo) y procesa colas si las hay.

6) Scheduler de acciones diferidas (cerrado)

Además de efectos, existen acciones “discretas” que deben ocurrir en un tick específico (spawn de evento, inicio de reforma, cambio de gobierno, etc.).

ScheduledAction:

id

run_tick: int

priority: int (mayor primero)

type: string

payload: JSON (o struct)

source: SourceRef

Ejecución: al inicio del tick, después de expirar efectos y antes de aplicar instantáneos, ejecuta todas las acciones con run_tick == tick en orden:

mayor priority

luego id lexicográfico

Estas acciones pueden:

mutar GameState directamente (cambios puntuales)

crear/expirar efectos

encolar nuevas acciones

Pruebas (qué garantiza esta definición)

Determinismo: mismo rng_seed + mismas decisiones + mismo orden → mismos resultados (sin drift de floats).

No hay ambigüedad en stacking ni en orden de aplicación.

“SET” funciona como bloqueo/override limpio (y evita combinatoria rara con ADD/MUL).

Cambiar contenido (eventos/reformas) no obliga a cambiar motor, solo a instanciar efectos/acciones con este contrato.
